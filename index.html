CH9

Virtual Memory

前一章提到的 paging 是技術，而這章提的是 paging 的使用方法。

Virtual Memory 就是使用到了 paging。

  • Progeam Size can grater than Phy Mem Size
  • Memory Utilization Up
  • CPU Utilization Up (since no need to load all program to mem)
  • Decrease I/O Transfer time

Demand Paging

要的時侯再 load

Locical Addr => Page Table query(invalid bit) => Page Fault => Fetch Lost Page in Disk to Mem Frame

Effective Memory Access Time:
(1-p)ma + p(Page Fault Process Time)

  • p: Page Fault Ratio

Page Replacement

Mem 換到 Disk

Page Replacement Algo:

  • FIFO
    先進先出
  • OPT
    理想
  • LRU
    No Belady
    需要硬體支援
    • Counter
    • Stack
  • LRU 近似
    退化成 FIFO, Belady
    • Second Chance
      With Reference Bit
      Clock pointer, clear bit until found 0
    • Enhance Second Chance
      Add Modify bit
      二進位數最小
  • MFU/LFU
    計算 ref 次數
    MFU: 最多的為 Victim
    LFU: 反之

Belady 現象

more frame, more P.F. ratio

分配頁框數議題

最少

取決於 Instruction Access Memory Count

IF DE FO EX WM
Must Maybe Maybe

Thrashing 震盪

解決方法：

  • 加入 P.F. Ratio 上下界
  • 降低 Multiprogramming Degree
  • Working Set Model
    依照 WSS 分配 Process 足夠的 frames
    若不夠 (Sum(WSSi) > Phy Mem) 則暫停某些 Process

Page Size 議題

Page Size ↓

  • Page Fault Ratio ↑
  • Page Table Size ↑
  • Whole Process I/O time ↑
  • Internal Fragmentation ↓
  • Single Page I/O time ↓
  • Locality 愈集中

管理 Free Space

Bit Vector

空 0
使用 1
EX: 0011001011000001
佔用 memory

Link List

效率不佳

Combination

使用可用區塊記錄其他可用區塊編號
效率不佳

Counting

使用可用區塊記錄其後可用連續區塊數
link list 在一起

Data Access Time

= Seek Time + Latency(Rotation 轉到磁頭) Time + Transfer Time
Seek Time 最久 因為是機械動作

Disk Scheduling Algo

FCFS (First Come, First Serve)

  • Simple
  • No Starvaion
  • 效果不佳

SSTF (short-seek time first)

  • Greedy 並非最佳
  • 可能 Starvaion

SCAN 雙向掃來掃去

  • 不必要的移動
  • No Starvaion

C-SCAN 單向掃

LOOK 雙向掃 遇到尾端請求折返

C-LOOK 單向掃 遇到尾端請求折返

Allocation Method

Contiguous Allocation

  • 分配連續空間
  • 檔案大小無法變動
  • External Fragmentation Problem

Link Allocation

  • list 存 block
  • 不支援 random access
  • Seek Time Long
  • Reliability 差

FAT (File Allocation Table)

  • 用 Table 存 link list 資料
  • EOF

Index Allocation

  • 檔案開頭放 Index Table 存每個 block index
  • No External Fragmentation Problem
  • File 大小可變
  • 無須先宣告檔案大小
  • Random/Sequential Access
  1. Index Block 可以自成 Link list
  2. Multilevel Index
  3. Unix 的 i-Node Structure. 有 15 pointers, 1~12 data block, 13 single level index, 14 two-level index, 15 three level index

Striping RAID
效能導向 平行讀寫 炸一顆全炸

RAID 1

Mirror RAID

RAID 0+1

RAID 0+1
至少四顆

成本高

RAID 1+0

RAID 2

Hamming Code 編碼 拆成位元單位
4 bits 7 部硬碟

RAID 3

  • parallel with parity
  • bitwise

RAID 4

  • Like 3, block wise

RAID 5

  • Like 4, block 切 bitwise 存

RAID 6

  • Double Parity
  • 壞兩顆也能修復

項目起源

回鍋 COC 有好一陣子了，在 PaD 的坑結束後過一陣子乏力期，一打開 IDA 就全身酸軟。

五月初又是一陣期中報告，實習面試，雖然一直有想要拆開它的慾望，卻一直抽不出身

就在考多益的前一週突然渾身不對勁的跳坑開始著手研究

研究過程

準備

COC 和 Pad 不同，是走 raw socket, 這様窺探資料的難度增加了 (這年頭沒有明文傳輸了吧)

所以也不需要 wireshark，先用 apktool 解開 apk

翻了翻 smali 果然重點都不在 java 層, 將唯一的 lib libg.so 丟到 IDA。一看太好了，沒有加殼！

開 strings subview 找到伺服器和 port

使用 python 開 tcp socket proxy 準備中間人解密資料

要讓 app 改連 proxy 首先想到 /etc/hosts 但改了沒有用而且不通用(需要 root)，後來是看到有現成的 fakedns，接著只要手動設定 wifi dns 地址即可。

之前弄的逆向相關項目幾乎都是用 C 實作，這次來玩玩看聽說處理底層資料也游刃有餘的 python 2
